Builders' Stack

A way to build software with a team of AI agents. Expensive models think, cheap models type, and files remember everything — so any session, any person, any model can pick up exactly where the last one stopped.

1The big idea

One model doing everything is expensive and forgetful. When the chat window fills up or the session crashes, the thinking is gone and you start over.

Builders' Stack splits the work and writes it all down:

  • Smart models (Opus / Fable) make the decisions — the spec and the plan. This is where deep thinking pays off.
  • Fast models (Sonnet) do the typing and the review — given a clear plan, they're quick, cheap, and reliable.
  • Files — not chat memory — hold every spec, plan, decision, and bit of progress. The chat can die; the work survives.

2The flow, end to end

idea→ /pm→spec→ /plan→plan→ /build→code→ /qa→verdict

Opus writes the spec and plan · Sonnet builds and reviews · if anything is unclear, it asks you instead of guessing.

3Three lanes — pick by how clear the task is

You don't run the full pipeline for every little thing. The work is spent where the uncertainty is. Just describe your task — the session picks the lane and tells you which one in a single line.

Fix lane XS · mechanical

Clear target, no decisions to make. A typo, a broken link, a bumped dependency.

/fix <description> → a Sonnet builder goes straight to work, reviewer checks after. No spec, no plan document.

Build lane S/M · the everyday default

You know what you want, it just needs designing and doing. Most coding lives here.

/plan <description> → Opus writes a short spec + plan in one step (no separate PM stage), Sonnet builds it, Sonnet reviews it.

Product lane ambiguous · large

The "what" itself needs deciding, or the task is big. New features, reworks.

/pm <idea> → Opus writes a proper spec and asks you the open questions first, then plan → build → review.

Lighter lanes drop documents, never discipline — every lane still gets its own branch, per-step saves, tests, and a review. The moment a small task turns out to need a real decision, it moves up a lane instead of guessing.

4The commands

CommandWhen you use itWhat happens
/adoptAdding this to a codebase that already existsOpus reads your code and drafts the context files; you confirm one batch of questions.
/fixTiny mechanical changeBuilder does it directly; reviewer checks.
/planA clear feature or changeOpus plans it (mini-spec inline), ready to build.
/pmA fuzzy or big ideaOpus writes a spec and asks you the open questions.
/buildA plan existsSonnet implements each step, saving after every one.
/qaA build is doneSonnet reviews it against the spec; gives a verdict.
/shipHands-off — do the whole thingPicks the right lane and runs it start to finish.
/statusAnytimeOne report: every task, who owns it, what's next.

Forgot a command? Just say what you want in plain words — the session names the right one and gets going. Commands speed you up; they're never a gate.

5Across the life of a project

The same handful of commands carry you from an empty repo to long-term upkeep. Only the mix changes.

Stage 0 · Bootstrap (once)

Make the context real before any planning — agents plan badly against blanks.

# new repo: fill in .claude/context/ by hand
/adopt our Django + Postgres API, tests via pytest   # existing code: derive it, confirm one batch
/status                                          # check the board is clean

Stage 1 · Zero to one (define what you're building)

The "what" is fuzzy, so you live in the Product lane. Decisions get captured while they're still cheap to change.

/pm a referral system that rewards inviter and invitee
        → answer its questions; the spec lands approved
/plan 2026-06-12-referral-system
/build 2026-06-12-referral-system
/qa 2026-06-12-referral-system
# or do it all at once:  /ship a referral system that rewards both sides

Stage 2 · Build-out (most of the project's life)

The product is understood; features are clear. Build lane is your default — skip the PM step.

/plan add pagination to the offers endpoint
/build 2026-06-12-paginate-offers
/qa 2026-06-12-paginate-offers
/fix offers endpoint 500s on an empty cursor       # quick bug → builder direct
/status                                          # end of session: what's in flight

Stage 3 · Scale & collaboration (more hands)

Several people or agents at once. The commands don't change; claims and /status handle the coordination.

/status                                          # who owns what; which claims went stale (>2h)
/plan migrate sessions table to partitioned storage  # declares the files it touches
/build 2026-06-13-partition-sessions             # overlapping work waits; the rest runs in parallel
# stale claim? read progress, /build picks up from the last good step

Stage 4 · Maintenance & hardening (mature)

Mostly small mechanical work, plus the odd decision that comes back around.

/fix bump axios past the advisory, redo the lockfile  # the workhorse here
/fix correct the rate-limit header name
/pm  do we still need the v1 webhook?                # a real question → back to Product lane
/status                                          # archive done tasks, keep the board small

Stage 5 · Handoff & onboarding (anytime)

Bringing in a person or a fresh agent is two files, not a tour.

# point them at: README → docs/one-pager.md → CLAUDE.md
/status                                          # the whole state of the world in one read
Rule of thumb: early project → more /pm. Mature project → more /fix and /plan. Reaching for /pm on every small task? The product's clear enough to drop to the Build lane. /fix keeps hitting design choices? You're under-specced — go up a lane.

6What you get

  • Nothing gets lost. Every step is saved and pushed — a crash costs you one step, not a day. Resume by reading one file.
  • It's cheaper. Most of the work (the typing) runs on the fast cheap model — roughly 3–5× less than letting a top model do everything.
  • Things run in parallel. Independent steps get built at the same time by separate agents.
  • A second pair of eyes, always. Review is built into the loop, with a cap so it can't spin forever.
  • You can answer "why." Specs, decisions, and even the agents' own logged mistakes live in files — six months later it's a lookup, not a guess.

7The three rules that matter most

  1. Files over memory. If the next session needs to know it, it goes in a file.
  2. The board never lies. The task list updates in the same save as the work it describes.
  3. Ask before you assume. Real questions go to you; small calls get made carefully and written down.
