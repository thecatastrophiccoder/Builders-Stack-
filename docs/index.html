Builders' Stack

Build software with a team of AI agents.

Expensive models think, cheap models type, and files remember everything — so any session, any person, any model picks up exactly where the last one stopped.

Hey, from Anmol 👋 — this is my experiment in running a real product workflow on a team of AI agents. I bring the ideas and the taste; they bring the planning and the typing. Everything they do is written down in files you can read — including their own mistakes. Steal whatever's useful, and let's keep making it better together — every contribution sharpens it.

The big idea

One model doing everything is expensive and forgetful. When the chat window fills up or the session crashes, the thinking is gone and you start over. Builders' Stack splits the work and writes it all down:

  • Opus / Fable — the smart models make the decisions: the spec and the plan. This is where deep thinking pays off.
  • Sonnet — the fast model does the typing and the review. Given a clear plan, it's quick, cheap, and reliable.
  • Files, not chat memory — every spec, plan, decision, and bit of progress lives in committed files. The chat can die; the work survives.

The flow, end to end

idea→ /pm→spec→ /plan→plan→ /build→code→ /qa→verdict

Opus writes the spec & plan · Sonnet builds & reviews · if anything's unclear, it asks you instead of guessing.

Three lanes — pick by how clear the task is

You don't run the full pipeline for every little thing. Effort is spent where the uncertainty is. Just describe your task — the session picks the lane and tells you which in one line.

Fix lane XS · mechanical

Clear target, no decisions. A typo, a broken link, a bumped dependency.

/fix <description> → a Sonnet builder goes straight to work, reviewer checks after. No spec or plan doc.

Build lane S/M · everyday default

You know what you want; it just needs designing and doing. Most coding lives here.

/plan <description> → Opus writes a short spec + plan in one step, Sonnet builds & reviews.

Product lane ambiguous · large

The "what" itself needs deciding, or the task is big. New features, reworks.

/pm <idea> → Opus writes a real spec, asks you the open questions, then plan → build → review.

Lighter lanes drop documents, never discipline — every lane still gets its own branch, per-step saves, tests, and a review. A small task that turns out to need a real decision moves up a lane instead of guessing.

What you get

Nothing gets lost

Every step is saved and pushed — a crash costs one step, not a day. Resume by reading one file.

It's cheaper

Most of the work (typing) runs on the fast model — roughly 3–5× less than a top model doing everything.

Runs in parallel

Independent steps get built at the same time by separate agents.

You can answer "why"

Specs, decisions, even the agents' logged mistakes live in files. Six months on it's a lookup, not a guess.

What people use it for

Different situation, same machinery — only the command mix changes.

🌙 Solo side project

Building across scattered evenings. Every session resumes from the index, not your memory. /status, then keep building.

👥 Small team + agents

Two humans and a handful of agents on one repo. Claims and per-step pushes stop collisions; takeovers are protocol, not Slack archaeology.

🏗️ Existing codebase

Dropping agents into a three-year-old project. /adopt reads the code and drafts the context pack; in-flight work onboards as lanes touch it.

💸 Token budget matters

Stop paying frontier prices for typing. Lanes route execution to the cheap model; the expensive one only thinks.

🧩 Refactor too big for one window

The plan is the memory. Builders execute steps across many sessions, losing at most one step to a crash.

📜 Audit & accountability

"Why is it built this way?" six months later is an ADR lookup in decisions.md, not archaeology.

Not a fit for: throwaway scripts and one-shot experiments — the ceremony isn't worth it. That's exactly what /fix (or just coding freehand) is for. The stack pays off the moment work outlives one sitting or involves more than one pair of hands.

How to use it — a real mini-app, start to finish

Let's build Snippy, a tiny URL shortener: paste a long link, get a short code; visiting the code redirects and counts the click. Here's the whole loop, command by command, and what each one produces.

0

Set up the context (once)

New repo: fill the TODOs in .claude/context/. Existing one: let Opus derive them.

/adopt Node + Express + SQLite service, tests via vitest
   → Opus drafts product/architecture/conventions, asks one batch of confirm questions
1

Describe the idea — Opus writes the spec

/pm a URL shortener — POST a long url returns a short code, GET the code redirects and counts clicks

It writes spec.md and comes back with the decisions it can't make for you:

# spec.md (excerpt)
Problem:  sharing long URLs is clumsy; we want short, trackable links.
Scope:    POST /shorten, GET /:code  ·  click counter per code
Non-goals: custom aliases, expiry, user accounts (not this version)
Acceptance:
  [ ] POST /shorten {url} → 201 {code, shortUrl}
  [ ] GET /:code → 302 redirect to original, counter +1
  [ ] invalid url → 400 ; unknown code → 404
Open questions:
  Q1 code length? — recommend 7 chars base62
  Q2 reject duplicate long urls or always mint new? — recommend reuse existing

You answer Q1 & Q2 (a quick tap), the spec is marked approved. The "what" is now locked.

2

Plan it — Opus turns the spec into steps

/plan 2026-06-12-url-shortener

It writes plan.md — interfaces first, then small green steps:

# plan.md (excerpt)
Files: src/store.js, src/codes.js, src/server.js, test/*
Step 1  codes.js — base62(7) generator + collision retry   verify: vitest codes
Step 2  store.js — save(url)->code, resolve(code), bump()   verify: vitest store
Step 3  server.js — POST /shorten, GET /:code, error paths  verify: vitest api
Decisions made (builder must NOT revisit): reuse existing url; 404 JSON shape

Notice: names, signatures, and edge cases are decided here, so the builder never has to guess.

3

Build it — Sonnet executes, saving every step

/build 2026-06-12-url-shortener
# commits land one per step, pushed immediately
step 1: base62 code generator + collision retry [url-shortener]
step 2: url store with reuse + click counter   [url-shortener]
step 3: shorten + redirect endpoints           [url-shortener]

If a step hits a real design gap, the builder stops and routes it back to Opus — it never improvises architecture.

4

Review it — Sonnet checks against the spec

/qa 2026-06-12-url-shortener
# review.md verdict
Verdict: approve-with-nits
AC1 POST/shorten  ✓   AC2 redirect+count ✓   AC3 invalid/unknown ✓
Tests: 11 passed   ·   Nit: extract base62 alphabet to a constant

A capped fix loop (max 2 rounds) handles "request-changes"; then it's yours to merge via PR.

The shortcut: all of steps 1–4 in one go — /ship a URL shortener with click counts. It triages the lane and runs the whole thing, stopping only to ask you the open questions.

Same commands, across a project's life

StageYou mostly reach for
Bootstrap (once)/adopt or hand-fill context, then /status
Zero → one (define it)/pm → /plan → /build → /qa
Build-out (most of the life)/plan for features, /fix for bugs
Scale & collaborate/status + claims; parallel /builds
Maintenancemostly /fix; /pm when a decision resurfaces

Rule of thumb: early project → more /pm; mature project → more /fix and /plan.

Cheat sheet

Commands

CommandWhenWhat happens
/adoptAdding the stack to an existing codebaseOpus reads the code, drafts context files, asks one confirm batch.
/fix <desc>Tiny mechanical changeBuilder does it directly; reviewer checks.
/pm <idea>Fuzzy or big ideaOpus writes a spec, asks the open questions.
/plan <id|desc>A clear feature/changeOpus plans it (mini-spec inline if no spec).
/build <id>A plan existsSonnet implements each step, saving after every one.
/qa <id>A build is doneSonnet reviews against the spec; gives a verdict.
/ship <idea>Hands-off, whole thingPicks the lane and runs it end to end.
/status [id]AnytimeOne report: every task, owner, what's next.

Forgot a command? Just say what you want in plain words — the session names the right one and gets going.

Pick a lane in 5 seconds

Mechanical, no decision?           → /fix
Clear "what", needs designing?     → /plan   (the default)
"What" is fuzzy, or it's big?      → /pm
Just do the whole thing for me     → /ship

Task statuses

backlog → spec → planned → building → review → done
                              ↳ blocked (side-state; back to building once Opus resolves it)

Build-lane tasks start at planned; Fix-lane tasks start at building.

Where things live

Best practices

Most of these are scar tissue — each one is in here because skipping it cost something once.

  1. Fill the context before the first real task. An Opus plan over an empty architecture.md is an expensive guess. Planning quality is capped by context quality.
  2. Keep context short and current. A wrong "fact" poisons every future plan. Prune ruthlessly.
  3. Problem before solution in specs. Acceptance criteria must be checkable by someone who read nothing else. Non-goals are what stop scope creep.
  4. Pre-answer the builder's decisions. Names, signatures, error handling, edge cases — every omission is a coin-flip handed to a cheaper model.
  5. Every plan step ends green and committed. Bisectable history, safe aborts, cheap takeovers.
  6. Recurring re-decisions mean a missing ADR. Write it once in decisions.md and stop paying for it.
  7. Cap autonomous loops. Two fix→review cycles, then a human. A loop that won't converge is a thinking problem, not a typing one.
  8. Watch the signals in /status. Many blocked → plans too thin (give Opus more context). Many request-changes → acceptance criteria too vague (fix the spec, not the builder).
  9. Archive done tasks. Move them to the index's Archive table so the live board stays readable in one glance.
  10. Commit .claude/ always. Specs, ADRs, and the index are project history, not scratch space.

The three rules that matter most

  1. Files over memory. If the next session needs to know it, it goes in a file.
  2. The board never lies. The task list updates in the same save as the work it describes.
  3. Ask before you assume. Real questions go to you; small calls get made carefully and written down.

Let's keep improving this — together

This workflow gets better every time someone uses it and notices something. That's not a nice-to-have; it's built into how it runs. Your contribution counts whether you're a human or an agent.

1 · Log what fought you

Hit a rule that got in the way, a thin plan, a case nobody anticipated? Append it to .claude/state/learnings.md — facts, no blame, with a concrete suggested fix. The agents do this on themselves; so should you.

2 · Turn lessons into rules

Good learnings graduate into ADRs in decisions.md or edits to the templates and conventions. A lesson learned twice is a process bug.

3 · Open a PR

Change a rule, a command, an agent — via PR with an ADR entry. Don't quietly ignore a rule you think is wrong; change it in the open so the next person sees why.

4 · Keep the docs honest

If behaviour changes, this page, the README, and the one-pager change with it. Stale docs are worse than none.

The whole bet of Builders' Stack is that writing things down compounds. Every spec, every logged mistake, every ADR makes the next task cheaper and the next contributor faster. Add yours.

How the safety net helps you

  • CI guards catch the common slips — a command name that clashes with a built-in, a code-map row pointing at a missing file, a leaked secret.
  • The reviewer is in every lane, so nothing reaches the default branch unread.
  • Per-step pushes + claims mean two people can contribute at once without stepping on each other.
